## Executive Summary

AEOS now provides a complete, read-only, local PR proposal management layer.
After generating a deterministic PR proposal (`aeos agent pr-proposal`), operators
can inspect and list proposals stored locally under
`~/.aeos/workspace/proposals/<id>/proposal.json` — without any GitHub API call,
without any git command, without any network request, and without any file mutation.

Two new commands complete the controlled PR lifecycle:

```
aeos agent pr list          — list all local proposals, sorted by date
aeos agent pr show <id>     — full detail of one proposal
```

Every output surface carries the invariant:
`read_only: true · applied: false · human validation required`

| Symbol | Role |

**`ProposalRepository.list()`** — silently skips invalid entries (one corrupt file
never breaks the full list). Sorted by `created_at` descending.

invalid status. Callers see a clean error, not a partial object.

## Proposal Storage Architecture

```
~/.aeos/workspace/proposals/
└── <proposal-id>/
    └── proposal.json
        ├── id           string
        ├── title        string
        ├── status       "pending" | "applied" | "dismissed"
        ├── created_at   ISO 8601 string
        ├── summary      string
        ├── files        list[string]
        └── diff_preview string | null
```

`ProposalRepository` reads this structure. It never writes to it.
The folder is populated by `aeos agent pr-proposal` (or manually by the operator).

## Controlled PR Lifecycle

```
MemoryRecord JSON files
       ↓
generate_pr_proposal_from_memory(name, memory_dir)
       ↓
PRProposal dataclass  →  aeos agent pr-proposal (text/json/file)
       ↓  (operator saves to workspace/proposals/<id>/proposal.json)
ProposalRepository.list()    →  aeos agent pr list
ProposalRepository.get(id)   →  aeos agent pr show <id>
       ↓
Human reviews, validates, creates PR manually
       ↓
read_only: true  ·  applied: false  ·  human validation required
```

At no stage does AEOS open a PR, push code, or apply any change autonomously.

## What the Controlled PR Layer Does NOT Do

By design. These constraints are permanent and enforced in code.

- Open a GitHub PR automatically
- Push code to any repository
- Apply any change to the project
- Call any LLM or AI API
- Make any network request
- Read `.env` files or environment secrets
- Write to `~/.aeos/projects.json`
- Modify existing `MemoryRecord` files
- Execute SQL against any database
- Deploy to staging or production
- Accept `applied: true` or `read_only: false` as valid state
