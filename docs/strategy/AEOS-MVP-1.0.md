**Recover · Govern · Migrate · Continue · Own.**

to governed, migration-ready, operable software — demonstrable in under an hour.

AEOS gives them the audit, the evidence, and the governance in under an hour.

that run the business but cannot be explained, modernized, or replaced without

## 4. Complete User Journey

The MVP delivers a complete, end-to-end experience — not a collection of commands.
Every step produces evidence. Every step is reversible. No step is destructive.

```
Step 1 — IMPORT            Import the project into AEOS
Step 2 — UNDERSTAND        Full project analysis: stack, providers, risks, sovereignty score
Step 3 — GOVERN            Automatic governance generation: all required documents, one PR
Step 4 — SECURE            Secrets stabilized, RLS hardened, .gitignore verified
Step 5 — PLAN MIGRATION    Migration roadmap generated, backup requirement documented, rollback SQL prepared
Step 6 — VALIDATE          Human reviews full migration plan — dry-run analysis, sovereignty delta, rollback path
Step 7 — READY             Migration readiness confirmed, evidence produced, MemoryRecord created
Step 8 — CONTINUE          Local AI development workflow activated — branch + PR discipline enforced
Step 9 — OWN               Periodic sovereignty audit, drift detection, memory timeline updated
```

### What the user experiences at each step

**Step 1 — Import**
The user points AEOS at a repository. That is all.
`aeos reclaim inspect --path <project>`
In 30 seconds: generator detected (Lovable/Bolt/Replit/etc.), providers mapped
(Supabase, Vercel, Firebase), control zones identified, initial sovereignty score.

**Step 2 — Understand**
`aeos reclaim harden --path <project>`
Full cross-domain audit: secret exposure, missing portability assets, RLS posture,
dependency risks, architecture unknowns. Risk-classified findings with remediation paths.
`aeos reclaim recovery plan` generates the complete 10-stage recovery roadmap.
Time: under 3 minutes.

**Step 3 — Govern**
`aeos build scaffold`
7 governance files generated and proposed as a PR:
ARCHITECTURE.md · docs/DECISIONS.md · docs/SECURITY.md · docs/SOVEREIGNTY.md ·
docs/AI-DEVELOPMENT-POLICY.md · aeos.toml · docs/features/RECOVERY.md
Human reviews and merges. Project moves from `weak` → `partial` maturity.
Time: under 5 minutes of human review.

**Step 4 — Secure**
RLS policies proposed, secrets rotation plan generated, .gitignore verified.
Each proposal is human-reviewed before any action is applied.
Evidence report produced at completion.
Project moves from `partial` → `controlled` maturity.

**Step 5–7 — Migration Readiness (when the user is ready)**
`aeos reclaim recovery plan` exposes stage_7 in full: source → target architecture,
backup requirement, dry-run analysis, rollback SQL prepared, sovereignty delta documented.
The human reviews the complete plan and validates the migration strategy.
No migration executes in MVP 1.0. The plan is the product.
`applied: false` — every output is a proposal, not an execution.

Migration execution (actual data movement and infrastructure switch) activates in V2,
once the readiness arc has been validated and trusted on real projects.

**Step 8 — Continue**
Local AI model configured in aeos.toml. Development continues with branch + PR discipline.
AEOS routes engineering tasks to local AI by default (free, private).
Frontier AI is available on explicit request, with context filtering and human approval.

**Step 9 — Own**
`aeos memory timeline --project <name>` shows the full progression.
`aeos reclaim harden` can be run again at any time — sovereignty score has moved.
The user can demonstrate sovereignty progress to any stakeholder.

## 7. The Workspace Role

### What it is

The Workspace is the graphical interface of AEOS — a local web UI that makes every AEOS
capability accessible without a terminal. It is the enterprise product.

### Why it is not in MVP 1.0

The CLI is the authoritative interface. It is:
- fully built
- fully tested (1555+ tests)
- deployable on any machine without installation overhead
- the most credible demo surface for a technical audience (CTOs, engineering leads)

For MVP 1.0, the CLI demo is the right demo. A senior engineer or CTO who sees
`aeos reclaim harden` produce a 200-line structured audit report in 30 seconds
understands the value immediately. No GUI required.

### What it becomes in V2

The Workspace unlocks the non-engineering buyer:

- A DSI who needs to see sovereignty progression without running commands
- A CEO who wants to show a bank that their project is production-ready
- A compliance team that needs audit trail evidence without terminal access

The Workspace is the **Series A product**. It is what turns AEOS from
a developer tool into an enterprise platform.

### Interface contract (defined now, implemented in V2)

The Workspace will expose:
- Project import (URL or local path)
- Stage progression dashboard (10 stages, current state, next recommended action)
- Sovereignty score history (timeline view)
- Evidence viewer (artifacts per stage)
- Migration readiness panel (stage_7 status, backup state, rollback path)
- Memory timeline (per-project audit history)
- PR review queue (governance files awaiting human approval)

All Workspace actions invoke CLI engine behavior. The Workspace is a display and approval
surface — not a separate execution path.

## 8. The Transformation Engine Role

### What it is

The Transformation Engine handles moving a project from one architecture to another —
controlled, tested, and reversible. It covers stage_7 of the recovery arc:
migration readiness, execution, and post-migration verification.

### In MVP 1.0 — Migration Readiness (not execution)

**Planning and readiness is fully included.** `aeos reclaim recovery plan` exposes stage_7 completely:
backup requirement, dry-run analysis, rollback SQL, migration sequence, sovereignty delta.
The plan is generated, human-readable, and auditable.

**What MVP 1.0 delivers for migration:**

- Complete migration plan: source → target, every dependency documented
- Backup requirement assessment: what must be backed up before any data action is authorized
- Dry-run analysis: schema comparison, data validation, risks enumerated — without touching production
- Rollback SQL: prepared and reviewed before any commitment is made
- Human validation gate: the human explicitly approves the migration strategy and risk posture
- Evidence report: migration readiness documented and referenced in MemoryRecord
- Sovereignty delta: before/after sovereignty level projected — Level 1 today, Level 4 after migration

**What is NOT in MVP 1.0 — execution:**

Migration execution (actual data movement, schema apply, infrastructure switch) is V2.
This is a deliberate product decision, not a technical constraint.

The reason: MVP 1.0 must first validate that the planning arc is accurate, trusted, and valued
on real projects. The human who reviews a migration plan in MVP 1.0 is the same human
who will authorize its execution in V2. The trust is built in MVP 1.0.

No migration executes in MVP 1.0. `applied: false` is the invariant.

**What "migration" means in this context:**
- Lovable Cloud → own Supabase Cloud account (Level 2 sovereignty)
- Lovable Cloud → self-hosted Supabase or PostgreSQL (Level 4 sovereignty)
- Firebase → PostgreSQL
- Supabase Cloud → on-premise or sovereign cloud

### Why this matters for the investor demo

Migration readiness is the highest-value moment in the investor arc.
A startup that has a complete, reviewed, approved strategy to move its Lovable project
to its own infrastructure — with backup verified, dry-run analyzed, rollback SQL ready —
has crossed from "we don't know how to migrate" to "we know exactly when and how we will own this."

That transformation is demonstrable in Demo B in under 10 minutes.
The execution (actual data movement) follows in V2 — with the same human gate, the same evidence trail.
The investor sees that migration is not a cliff. It is a staircase with a railing.

## 9. The Continuation Engine Role

### What it is

The Continuation Engine manages the ongoing development of a recovered or newly created project.
After recovery and migration, the team continues building — with AI assistance that is
local-first, governed, and sovereignty-aware.

### In MVP 1.0

**AI routing is production.** `aeos.toml` configures local and frontier providers.
`aeos ai ask` routes to local by default. Frontier requires explicit authorization.
`require_human_approval = true` ensures no silent escalation.

**The guided development workflow is in MVP 1.0 scope:**
- Branch discipline enforced for every AI-assisted task
- Context filtering: secrets and sensitive data stripped before any frontier call
- Local AI for routine tasks (explain, test, refactor, document)
- Frontier AI for complex reasoning — with human approval and usage logging
- Every AI-assisted change produces a PR, not a direct commit

### The cost story

This is the most compelling story for African and emerging market teams:

| Task | Without AEOS | With AEOS |
|---|---|---|
| Explain a function | Frontier AI — $0.05–0.20 | Local AI — free |
| Write a unit test | Frontier AI — $0.10–0.30 | Local AI — free |
| Refactor a module | Frontier AI — $0.20–0.50 | Local AI — free |
| Architecture decision | Frontier AI — $0.30–1.00 | Frontier, human-approved |
| Debug complex regression | Frontier AI — $0.50–2.00 | Frontier, human-approved |

For a team doing 50 engineering AI interactions per day:
- Without AEOS: $15–50/day in frontier AI costs = $450–1,500/month
- With AEOS (local-first): $2–5/day for the 10–20% genuinely complex tasks

**The ROI of AEOS for a small team is measurable in weeks.**

## 13. Why Now?

### The timing is precise

In 2024, Lovable grew from zero to 2 million users in under 12 months.
Bolt, Replit, Cursor, and Claude Code collectively represent tens of millions of AI-assisted
software projects created in the last 24 months.

This is unprecedented. Never before in the history of software has so much code been
created so fast, by so many people, with so little engineering governance.

### The ceiling is arriving

The projects built in 2023–2024 are maturing. Their creators are pitching to enterprise clients.
They are hiring their first engineers. They are facing their first security audits.
They are applying to government procurement programs that require data residency.

**For the first time, the industrialization gap has become a commercial blocker — at scale.**

The market is not waiting for another AI generator.
The market is waiting for the layer that makes AI-generated software production-ready.

### The infrastructure gap has no incumbent

There is no product that does what AEOS does.

- Lovable, Bolt, Replit, and Windsurf generate code. They do not govern it.
- Cursor, Claude Code, and GitHub Copilot assist development. They do not audit sovereignty.
- Snyk and Trivy scan security. They do not produce governance arcs.
- Terraform and Pulumi manage infrastructure. They do not recover AI-generated projects.

**AEOS is not competing with any of these tools. It completes all of them.**
